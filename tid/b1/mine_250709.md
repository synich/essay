# 大模型5交互协议

## REST协议

早期协议百花齐放，有FastChat、TGI协议，到25年基本上都遵循OpenAI定义的JSON的HTTP协议了。GPT3时代，有`/v1/completions`协议，ChatGPT发布后，只要会话补全`/v1/chat/completions`就足够了。私有部署无认证，需要的话加Bearer。多轮对话要把问答双方的内容都放在messages数组中。如果轮数超出，保留第一个system，删除开始的user消息。

```
curl --request POST \
  --url https://api.siliconflow.cn/v1/chat/completions \
  --header 'Authorization: Bearer sk-arnuqrnsfupcdlyyirqelnvetjocwuqwwcwghymogdvipdql' \
  --header 'Content-Type: application/json' \
  --data '{
  "model": "Qwen/Qwen2.5-7B-Instruct",
  "messages": [
    {
      "content": "You are ChatGPT",
      "role": "system"
    },
    {
      "content": "你会说中文吗",
      "role": "user"
    }
  ],
  "tools": [],
  "stream": false,
  "max_tokens": 4096,
  "temperature": 0.8
}'
```

model参数必填，如果不知道，通过`/v1/models`获取。

如果stream是true，响应会改为SSE格式，收到 `[DONE]` 后，偶尔浏览器会再发一次POST，并得到关于token的usage。

### messages和role详解

第一条消息role用system，设置初始对话背景，目标和约束。之后提问者是user，回答者是assistant。

以上是经典的聊天过程，有了工具调用，就有了新的role: tool。tool和user等价，可以理解为user是人的提问，tool是工具的使用结果。user或tool都配合content字段带上内容。见过有些客户端发送tool会加name，但似乎没有name也并不影响。

有了工具后，assistant的响应就会区分content和tool_calls，二者通常互斥，但有些模型会两个都返回，这时客户端会显示content并执行tool_calls

在下一次请求里，历史messages的assistant部分要带上content和tool_calls数组，作为整个上下文的一部分，配合下一条通常是role: tool的执行结果(content字段)，构成一段完整的问答，模型如果判断工具执行成功，就会返回assistant的content和空的tool_calls，表示结束。如果assistant不带上tool_calls，大模型可能会觉得内容缺失，从而继续发tool_calls，导致客户端继续执行工具，从而陷入死循环。

## langchain和ollama交互

langchain是core，有很多的外围适配。多见的是langchain_llm，和ollama配套的则是langchain_ollama。langchain_core库的主要部分似乎是http库，整体大小只有数M。

两大流派

* ChatOllama(langchain_core.language_models.chat_models.BaseChatModel)
* OllamaLLM(langchain_core.language_models.llms.BaseLLM) 又叫oldfashion

必不可少的embed

* OllamaEmbeddings
