# 鉴权与API认证差异比较

大约2016年我刚做安全工作时，给嵌入式设备做的是chanllenge-respose登陆。这几天做了新的API认证，用到的新技术和思路和原来做的大为不同，但细想又有理可循。

方案是服务于问题的，只有定义好问题，方案才是合适的。嵌入式设备由于NetSDK的存在，最大的问题是要在一个明文网络上安全的发送密码，CR方案解决的就是这个问题。但这次要做的程序，已经具备了TLS，所以传输明文密码并不是问题，但怎么更方便的调用API才是这次的焦点。

## 从Bearer到JWT最后OAuth

以前做CR登陆，随着TLS的全面普及已经换了玩法，26年HTTPS已经成为事实的标准，认证当然选择Bearer格式。至于内容其实放一个固定字符串当然也可以，但既然设计总要玩点花活，这就推导出第二层规范：Bearer的内容格式，几乎是公认的JWT即Json Web Token标准。再次重申，由于有TLS的加持，密钥的加密已经不是考虑的重点，所以JWT的标准三段内容header/payload/signature没有特别的加密，最多就是有签名校验防篡改。

要怎么获取payload里的凭据？其实CR也可以，但CR要求服务端记录密码的hash值，一旦泄露就等于密码泄露，于是又引入了加盐机制，使彩虹表失去了意义。现代流行的加密做法是PBKDF2，由于有TLS的保护，用户登陆发送的直接就是明文密码，防泄露就成了服务端关注重点，所以服务端存的不再是密码hash，而是用PBKDF2（早先有Argon2）将明文密码＋盐做60万次hash，再保存下来。保存格式：`$pbkdf2-sha256$i=600000$salt$final_hash`。坏处就是登陆会比较耗时，计算过程可能要数十数百毫秒，但相应的破解的难度也极大的提升。

登陆通过后签发两个key，一个叫access key，一个叫refresh key。前者通常半小时，后者则几天，在安全性和计算强度上取折衷。JWT的签名则通过服务端的secret key保证，这个secret全局唯一，不和用户绑定，而且保存在文本里，启动时加载一次，这样每次验证HMAC就不需要读数据库，做到快速验证。由于有了PBKDF2机制，即使真的出现secret丢失，也不会导致密码泄露，但需要所有用户重新登陆配合secret生成新的访问key
