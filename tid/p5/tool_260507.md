# 终端与isatty

发现rg在终端的行为和管道的行为不一样，终端的结果会带颜色，而且会把同一个文件的检索结果合并，而管道则表现的和常规的grep类似。这引起了我的好奇，查阅之后知道因为rg会检测stdout是否tty，如果是则以人类友好的方式显示，当不是的场景会回落到传统模式，而检测的核心函数就是isatty。

进一步了解了isatty的实现原理，发现它竟然只是简单的调用了fstat然后判断是否是*字符设备*。显然字符设备的含义要广得多，除了终端，还有串口、音频、/dev/null、/dev/urandom都是字符设备，真正的终端支持ioctl(TIOCGWINSZ)获取宽高，还支持tcgetattr获取termios属性，包括波特率（虽然在网络时代只用于兼容，没有实际意义）、自动换行、按行缓冲等高阶属性。

为什么isatty只是简单地判断是否字符设备？在isatty函数诞生的年代，字符设备和tty可以划等号，之后众多的程序（尤其shell）都广泛且极为高频地使用了isatty，导致既不能改函数名，也不能修改实现增加耗时操作（ioctl和termios都很耗时），所以isatty只能是一个快速但不准确的判断，更精准的还依赖应用层额外调用ioctl。

查看self/fd，会发现tty情况下012都指向/dev/pts/0，而一旦以管道方式启动，1会指向pipe:[1356]，0和2仍指向/dev/pts/0。这是进程创建时通过dup2做的调整。增加了2>&1之后，2自然也就指向了pipe:[1356]。

澄清一个误区，块设备和字符设备不是非此即彼的关系，还有管道和套接字，以及更常见的文件、目录、软链接，它们在VFS层面都属于同一层级。
